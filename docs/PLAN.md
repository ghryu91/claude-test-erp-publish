# ERP 웹서비스 전환 계획 - 놓치고 있는 부분 분석

## Context
Delphi WinForm ERP → 웹 전환. 다중 서버/DB/테이블, 고객사별 커스터마이징된 스키마,
모기업-자회사 구조, React+TypeScript+Tailwind 프론트, 백엔드 분리 예정.
"내가 뭘 놓치고 있는지" 심층 분석 요청.

## 프로젝트 개요

Delphi로 만든 Windows ERP 프로그램을 웹으로 전환하는 프로젝트.

- **고객:** 레미콘/아스콘/건설자재/물류 업체
- **현재 구조:** 다중 서버(10개+), 다중 DB, 업체별 커스터마이징된 스키마
- **MAIN DB:** 업체 정보, 사용자 관리, 프로그램 접근 권한
- **업체별 DB:** 비슷하지만 컬럼명/추가 컬럼이 다름 (Cust_Code vs custcode 등)
- **모기업-자회사 구조:** A회사가 B,C,D 자회사 데이터에 접근 가능
- **목표:** 프론트엔드 하나로 통합, 고객별 기능 on/off로 맞춤 제공
- **기술 스택:** React + TypeScript + Tailwind CSS (프론트) / 백엔드 API 분리 예정
- **향후:** API를 통해 웹뿐 아니라 C# WinForm, Python PySide, 모바일 앱에서도 사용

### 작업 우선순위
1. 기존 프로그램의 read 기능 구현 (조회)
2. write 가능한 admin 페이지
3. Delphi 프로그램을 대체하는 종합 웹 프로그램

## 놓치고 있는 부분 분석

## 1. 가장 중요한데 언급 없는 것 — Multi-tenant 설계

현재 구조는 사실상 **SaaS Multi-tenant** 구조임.
고객사마다 DB가 다르고, 스키마도 다르고, 기능도 다름.
이걸 프론트에서 "하나처럼" 보이게 하려면 백엔드에 **추상화 레이어**가 필요.

```
[Frontend] → [API Gateway / BFF] → [고객사 A DB]
                                 → [고객사 B DB]
                                 → [고객사 C DB (자회사)]
```

BFF(Backend for Frontend): 프론트가 요청하면, 어느 서버/DB로 라우팅할지
백엔드가 알아서 처리. 프론트는 그냥 "데이터 주세요"만 하면 됨.

---

## 2. 스키마 차이 처리 전략 (Cust_Code vs custcode 문제)

놓치기 쉬운 핵심. 두 가지 방법:

**방법 A: 매핑 테이블** (MAIN DB에 저장)
- 고객사 ID + 필드명 + 실제 DB 컬럼명 매핑
- 백엔드가 쿼리 날릴 때 매핑 테이블 참조해서 동적 쿼리 생성

**방법 B: ORM + 추상 모델**
- 내부적으로 표준 필드명 정의
- 고객사별 어댑터 클래스가 변환 담당

→ 초기엔 방법 A가 현실적. 나중에 B로 발전 가능.

---

## 3. MS SQL 2000 문제 — 생각보다 심각

- ROW_NUMBER() 없음 → 일반적인 페이징 불가
- TOP N 방식으로 흉내는 가능하지만 성능/정확도 이슈
- 해결 방법들:
  1. **백엔드에서 전체 읽고 메모리 페이징** (데이터 적으면 OK)
  2. **가상 스크롤(Virtual Scroll)** — Delphi처럼 빠르게 보이는 효과
  3. **캐싱 레이어(Redis)** — 자주 쓰는 데이터 캐시
  4. **장기적으로 DB 마이그레이션 계획** 세워야 함

---

## 4. 모기업-자회사 컨텍스트 전환

단순히 "서버 바꾸기"가 아니라 설계가 필요:
- 세션에 현재 "컨텍스트(회사ID + 서버ID)" 저장
- 모든 API 요청에 컨텍스트 포함
- 프론트에서 "지금 어느 회사 데이터 보는 중"인지 명확히 표시
- 권한 체계: 모기업 사용자가 자회사 접근 시 어디까지 허용?

---

## 5. 기능 플래그(Feature Flag) 시스템

고객 요구사항 "이거 추가해주세요" 문제 해결책:
- 기능을 on/off 스위치로 관리
- 고객사 A는 GPS/RT 기능 있음, B는 없음 → DB에 설정값으로 관리
- 프론트에서 플래그 보고 메뉴/버튼 표시 여부 결정
- 버전 관리보다 훨씬 유지보수 쉬움

---

## 6. 컬럼 설정 저장 — localStorage만으론 부족

현재 localStorage → 기기/브라우저 바뀌면 초기화됨.
**설계안:**
- localStorage: 즉시 반응 (캐시 역할)
- MAIN DB 사용자 테이블: 진짜 저장소 (로그인 시 불러옴)
- 동기화: 변경 시 DB에도 저장 (debounce 처리)

---

## 7. Radix UI가 뭔지

오늘 처음 들었다고 했으니 설명:
- **기능은 있고 스타일은 없는 컴포넌트 라이브러리**
- Dialog, Select, Dropdown, Tooltip 등 복잡한 UI 로직을 직접 구현하지 않아도 됨
- 접근성(Keyboard 네비게이션, 스크린리더) 자동 처리
- Tailwind로 원하는 디자인 입힐 수 있음
- **shadcn/ui** = Radix UI + Tailwind 조합의 대표적 라이브러리

ERP에서 쓸만한 것들: Modal, Select, DatePicker, Tooltip, ContextMenu

---

## 8. 백엔드 기술 스택 결정 필요

C# 언급했는데, MSSQL 환경이면 **C# .NET Web API**가 가장 자연스러움:
- MSSQL 드라이버 성숙도 최고
- Entity Framework로 ORM
- 기존 Delphi 로직 이해하는 개발자라면 C# 진입장벽 낮음

Python FastAPI도 선택지:
- 빠른 개발, 간단한 문법
- MSSQL 연결 가능하지만 드라이버 설정 복잡할 수 있음
